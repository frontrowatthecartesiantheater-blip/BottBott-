# Autoblog Template Extraction Plan

Forward-looking notes for extracting a finished client build into this
reusable, client-agnostic template repo, and for keeping the template current
with design changes made in client builds after the template was last
generated. This is planning material, not the extraction itself.

## How this template relates to a client build

- **The client build (source repo)** — a deployed, client-specific instance of
  this pipeline. The source of truth for what gets extracted.
- **`autoblog-pipeline-template`** (this repo) — the reusable destination.
  Generated by genericizing a client build: client-specific values are replaced
  with `{{PLACEHOLDER}}` tokens and centralized in `lib/client-config.js`, and
  client-specific docs become blank `*-template.md` files.

The template is generated from a client build at a point in time. Any design
change made in a client build after that point must be re-applied when the
template is next regenerated, or the template will carry an abandoned design.

## A. Keyword architecture

The model below is the current keyword design and must be reflected in any
future extraction.

### Current design

- **`keywords` table** — `id`, `term` (unique), `tier` (`primary` | `secondary`),
  `created_at`.
- **`topic_keywords` join table** — `topic_id`, `keyword_id`, composite primary
  key `(topic_id, keyword_id)`, `on delete cascade`.
- Each topic keeps its own **`primary_keyword`** column on the `topics` table.
  Two **supporting keywords** are then associated per topic through
  `topic_keywords`. Association is **topic-level** (each topic gets its own
  supporting terms) rather than category-level.

This **replaces an earlier orphaned `keywords` table** that had
`keyword` / `topic_tags` / `priority` columns. That older table is the
abandoned design — do not carry it into the template.

### Seed data and script

- Seed data lives in two separate files, `content/seed/topics.json` and
  `content/seed/keywords.json`.
- A dedicated script, `scripts/seed-topics-keywords.js`, populates them —
  **separate from** the original generic `seed.js`. It handles literal dates and
  descriptive fields.

### Keyword-strategy document and placeholder convention

Each client build is informed by a keyword-strategy document: a tiered SEO
keyword list, typically a Tier 1 / Tier 2 / Tier 3 prioritization. In the
template this document is referenced through the **`{{CLIENT_KEYWORD_STRATEGY}}`**
placeholder, so each new client supplies their own.

The strategy doc's tiers map onto the `keywords.tier` column as follows:

- **Tier 1 → `tier: primary`**
- **Tier 2 and Tier 3 → both collapse into `tier: secondary`**

## B. Warning: regenerate `schema.sql` fresh, never copy

Any `schema.sql` snapshot shipped in the template **must be regenerated fresh
from the live migrations in `supabase/migrations/` at the start of each new
client build.** Never copy an existing static `schema.sql` file.

A stale snapshot predates the keyword rework in section A and would ship the
template with the already-abandoned keyword design (the orphaned
`keyword` / `topic_tags` / `priority` table). Regenerating from the live
migrations is the only way to guarantee the template reflects the current
schema.
